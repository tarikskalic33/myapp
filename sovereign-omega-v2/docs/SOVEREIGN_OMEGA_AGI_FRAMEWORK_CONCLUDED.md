## Part III — The Hardware Optimisation Stack

Archive 4 introduces a three-layer hardware optimisation system that is a mandatory
prerequisite for stable production operation on the AMD RX 570 / 8GB RAM. The layers
form a strict dependency chain.

**PGCS — Predictor-Guided Compression Swapping (T0 gating prerequisite)**

PGCS prevents OOM crashes and disk swapping by proactively compressing inactive
memory segments via integer quantisation, reducing the context window memory
footprint by up to 50x and keeping the entire active context inside physical RAM.
The compression engine triggers at 92% memory capacity. Success criterion is absolute
and binary: hardware-level disk page-ins and page-outs must equal exactly zero under
the full 12-hour stress load. If PGCS fails, all downstream telemetry is invalid.

**TGCS — Thermal-Gated Cycle-Stretching (T1 dependent on PGCS)**

Manages intense thermal throttling under sustained inference load by dynamically
stretching compute cycles when approaching thermal limits. Success criterion: telemetry
run-to-run variance sigma-squared drops to absolute zero.

**AFSE — Asymmetric Flux Scaling Engines (T1 analytical)**

Characterises how the system scales from consumer hardware to distributed cloud
topologies. Success criterion: correlation index R-squared >= 0.98 between performance
profiles on the AMD RX 570 and true distributed topologies. This is commercially
significant: it means the governance substrate validated on a 200 euro GPU generalises
to enterprise-grade infrastructure.

Archive 4 introduces three breakthrough vectors that extend the Python core matrix

**Vector 1 — Unprivileged eBPF Fallback (T1, container-safe)**

**Vector 2 — Epoch Failsafe Isolation (T0, cascade resilience)**

falls back to an immutable ledger state verified by quick-consensus. Success criterion

correction, not learning. Success criterion: dynamic quantisation drift index D = 0
